## 0. 이 문서를 읽는 법 (신입용 안내)

처음 온 분에게:
- 우리는 **SecureAI**라는 AI 보안 감사 플랫폼을 만듭니다. 코드를 AI로 분석해서 보안 취약점을 찾고(SAST), 실제로 공격해보고(DAST), 자동으로 고치는(패치) 제품이에요.
- 기술은 이미 동작합니다. **문제는 "사업으로서 방어 가능한가"입니다.** 한 투자 심사역(VC)이 우리를 보고 "기술은 됐는데, 따라잡히기 쉽고, 검증이 없고, 너무 넓다"고 지적했어요.

## 1. 배경 — VC가 지적한 5가지 (요약)

| # | 지적 | 한 문장 요약 |
|---|------|------------|
| ① | **해자(moat)가 없다** | 핵심 로직이 "코드를 잘라 Claude에 던지고 답을 파싱"하는 래퍼다. 며칠이면 복제된다. |
| ② | **레드오션이다** | Snyk·CodeQL·Semgrep·Copilot Autofix가 이미 IDE/PR에 들어가 있다. 우린 후발주자다. |
| ③ | **신뢰성 검증이 없다** | 오탐율(FP)·탐지율(recall) 숫자가 없다. LLM 할루시네이션·잘못된 패치 가드가 없다. |
| ④ | **범위가 너무 넓다** | 도메인 20개. "하나도 1등이 아니다"라는 신호. 날카로운 쐐기(wedge)가 없다. |
| ⑤ | **단위 경제성이 위험** | 스캔마다 토큰 비용이 직접 나간다. "받는 돈 > 토큰 원가"가 증명 안 됐다. |

> 핵심 통찰: **AI 앱의 해자는 모델이 아니라 "모델 주변"에서 나온다.** 모델은 빌려 쓰는 부품이라고 인정하고, (1) 데이터 플라이휠, (2) 워크플로/증적 락인, (3) 검증 레이어 — 이 셋에서 싸운다.

## 2. 한 장 전략 — "쐐기는 이미 코드 안에 있다"

우리는 **CISO 보고서 · 행안부 체크리스트 · ISMS-P 증적 PDF 자동 생성**을 이미 만들어 놨습니다(Sprint 9). 그런데 이걸 "부가 기능"으로 묻어놨어요. **이게 제품 본체여야 합니다.**

> ### 🎯 새 포지셔닝 한 문장
> **"한국형 컴플라이언스 증적 자동화 + 동적 검증(DAST) 보안 플랫폼"**
> — 범용 SAST로 Snyk와 싸우지 않는다. **글로벌 제품이 약한 한국 규제(ISMS-P·행안부·전자금융·개인정보보호법) 증적 자동화**라는 쐐기로 좁힌다. 그리고 "찾았다"가 아니라 **"샌드박스에서 털어서 증명했다"**로 신뢰를 만든다.

왜 이게 합리적인가 (신입에게):
- Snyk·GitHub는 **한국 규제 증적을 안 만들어 줍니다.** 언어 + 규제 + 로컬 감사관행이라는 3중 장벽 = 복제가 느린 해자.
- 구매자가 다릅니다. SAST는 "개발 예산"의 5번째 스캐너지만, **컴플라이언스 증적은 "감사 예산"에서 사고, 매년 강제로 해야 하고, 지금은 사람이 엑셀로 합니다.** "안 하면 인증 떨어지는 일" = 교과서적 쐐기.
- 판매 논리가 "취약점을 더 잘 잡습니다"(증명 어려움) → **"ISMS-P 준비 3주를 3일로 줄입니다"**(증명 쉬움, 예산 명확)로 바뀝니다.

나머지 19개 도메인(auth/org/team/credit…)은 **지우는 게 아니라 피치에서 숨깁니다.** 컴플라이언스 제품을 굴리기 위한 하부구조일 뿐.

## 3. EPIC-VAL — 신뢰성 검증 (지적 ③ 대응) 🔴 최우선

> **왜 1순위인가:** 보안 제품에서 "숫자 없음"은 생존 문제다. VC에게 보여줄 첫 슬라이드가 "FP율 X%, 탐지율 Y%"여야 한다. 숫자가 없으면 나머지 논의가 무의미하다.

**목표:** 공개 벤치마크 기준 정량 점수를 만들고, CI에서 자동 회귀 검사한다. 할루시네이션·패치 안전 가드를 코드로 넣는다.

| **TASK-1302** | 평가 결과를 CI 게이트로 연결 | 프롬프트/모델 변경 시 점수 자동 측정. 기준점 대비 하락하면 CI 경고 | M |

**목표:** 컴플라이언스 증적 자동화를 제품 본체로 끌어올리고, 나머지를 하부구조로 재배치한다.

## 8. FAQ (신입이 궁금해할 것들)

**Q. 도메인 20개를 다 버리나요?**
아니요. **코드는 그대로 두고, 외부에 말할 때만 "컴플라이언스 증적 자동화" 하나로 좁힙니다.** auth/org/team 등은 그 제품을 굴리는 하부구조예요.

**Q. 왜 한국 컴플라이언스죠? 글로벌이 더 크지 않나요?**
글로벌 범용 SAST는 Snyk가 이미 1등이라 우리가 못 이깁니다. 한국 규제 증적은 **글로벌 제품이 안 하고(언어·규제 장벽), 매년 강제로 사야 하는 시장**이라 후발주자도 1등이 될 수 있어요. "이길 수 있는 작은 전장"이 핵심.

**Q. LLM이 틀린 취약점을 보고하면요?**
그게 TASK-1303(AST로 인용 검증)과 TASK-1304(DAST로 실제 익스플로잇 증명)가 막는 부분이에요. **"추측을 증명으로 바꾸는 레이어"가 우리 차별점**입니다.

**Q. 모델(Claude)이 우리 해자라는 거 아니었나요?**

## 부록 A. 용어집 (신입 필독)

| 용어 | 뜻 |
|------|-----|
| **SAST** | 정적 분석. 코드를 실행하지 않고 읽어서 취약점 찾기 |
| **DAST** | 동적 분석. 실제로 공격(익스플로잇)을 날려서 취약한지 확인 |
| **FP (False Positive, 오탐)** | 취약점이 아닌데 취약하다고 잘못 보고한 것. 많으면 신뢰를 잃음 |
| **Recall (재현율, 탐지율)** | 실제 취약점 중 우리가 잡아낸 비율 |
| **Precision (정밀도)** | 우리가 취약하다고 한 것 중 진짜인 비율 |
| **OWASP Benchmark** | 보안 스캐너 성능을 재는 표준 시험지(Java, 2,740 케이스). 점수 = TPR − FPR |
| **Moat (해자)** | 경쟁사가 따라오기 어렵게 만드는 방어벽 |
| **Wedge (쐐기)** | 처음 시장에 박고 들어가는 좁고 날카로운 진입점 |
| **Data Flywheel (데이터 플라이휠)** | 쓸수록 데이터가 쌓이고, 그게 제품을 좋게 만들어 더 많이 쓰게 되는 선순환 |
| **System of Record** | 고객의 핵심 데이터가 "사는 곳". 여기 데이터가 쌓이면 못 떠남 |
| **BYOK (Bring Your Own Key)** | 고객이 자기 AI API 키를 가져와 쓰는 것 |
| **Unit Economics (단위 경제성)** | 거래 1건당 "버는 돈 − 쓰는 비용". 양수여야 사업이 됨 |
| **ISMS-P** | 한국 정보보호 및 개인정보보호 관리체계 인증. 많은 기업이 매년 받아야 함 |
| **증적(Evidence)** | 감사관에게 "우리 이렇게 보안 지켰어요"를 증명하는 자료 |
