## 0. 한 줄 요약 (정직 버전)

> **OWASP Benchmark(Java) 961케이스 단일 패스 측정 결과, 실제 익스플로잇 가능한 인젝션 계열(SQLi·CmdI·XPath·Path Traversal·XSS·LDAPi)에서 탐지율 84.6%를 기록했다. 전체 평균 탐지율(43.9%)은 단일 패스 LLM SAST가 구조적으로 약한 암호·설정 위생 계열이 끌어내린 값이다.**

숫자를 부풀리지 않는다. 강점(인젝션 탐지)과 약점(오탐율·암호 계열·모델/방법론 한계)을 같은 슬라이드에 둔다. VC가 묻는 건 "정직한 측정 체계가 있는가"다.

### 3.1 계열 합산

| 계열 | 탐지율 | 오탐율 | 정밀도 | score | TP/FP/TN/FN |
|------|-------:|------:|------:|------:|-------------|
| **인젝션 6종** (실제 익스플로잇 위험) | **84.6%** | 49.6% | 66.6% | 0.350 | 225/113/115/41 |

## 4. 신뢰 레이어 — "추측을 증명으로" (정성, 차별점)

숫자 외에 VC가 사는 건 **검증 서사**다 (doc 18 §3, ③ 대응).

- **VAL-3 결정론적 AST 가짜인용 가드** (Sprint 13): LLM이 보고한 `file:line`이 실제 코드에 존재(파일 범위 내·비주석/비공백)하는지 **AST로 검증 → 불일치 findings 자동 폐기**. 저장되는 findings는 **모두 라인 실재 = "가짜인용 0건" 구조적 보장**. 폐기 건수는 런타임 metric으로 노출(=차단한 할루시네이션 수).
  - ⚠️ 정직 표기: "0건 보장"은 구조적(설계상)이며, **벤치마크 전반의 집계 폐기 수·검증 후 FPR 개선폭은 아직 미측정**. → §5 다음 단계.
- **VAL-1 평가 하니스 + VAL-2 CI 회귀 게이트**: 프롬프트/모델 변경 시 점수가 자동 측정·비교되어 **"숫자가 관리되는 체계"**가 코드로 존재. (단발 자랑 숫자가 아니라 회귀로 지켜지는 숫자.)
- **SAST→DAST `proven_exploitable`** (Sprint 14 이월 VAL-4): SAST 의심 → DAST 샌드박스 실제 익스플로잇 성공 시 라벨 부여. **아직 미구현** — "증명했다" 데모는 로드맵(7~8주차).

## 5. 한계 — 먼저 말한다 (VC 신뢰 확보)

1. **오탐율 27%(인젝션 50%)는 높다.** 이게 다음 우선순위. 줄이는 수단: VAL-3 AST 가드(가짜인용 제거) + MOAT-1 트리아지 피드백 리랭커 + DAST 증명. **"검증 후 FPR"은 아직 측정값 없음.**
2. **"recall 80%에서 FPR" 산출 불가.** 단일 임계 분류라 ROC 스윕 불가 → 신뢰도 점수 출력 + 임계 튜닝이 후속 과제. 현재 가장 정직한 운영점은 인젝션 계열(recall 85% / fpr 50%).
3. **헤드라인이 비-기본 모델(gemini-2.5-flash)에서 나왔다.** 제품 기본은 Claude. §6.
4. **암호·설정 위생 계열 탐지율 ~0%.** 의도적(룰 영역)이라 설명하되, "전 영역 SAST"로 과대 포장 금지.
5. **샘플(961) ≠ 풀런(2,740).** 절대 수치는 풀런 재확인 대상.

## 6. 의사결정 필요 — 헤드라인을 Claude로 재측정할까?

대표런은 원가/속도 때문에 **gemini-2.5-flash**로 돌렸다. 그러나:
- 제품 기본 모델은 **Claude**(doc: 최신·최강 모델 우선).
- VC 슬라이드 숫자는 "우리가 실제 파는 구성"에서 나와야 신뢰가 간다.

→ **권고: 헤드라인 숫자는 Claude(현행 기본)로 LIMIT=100 1회 재측정** 후 본 문서 §2·§3 갱신 + `baseline.json` 모델 표기 정정 + kkebi README 검증 숫자 동기화. (gemini 결과는 "저비용 모델 대조군"으로 병기하면 원가 서사에도 활용 가능.)
