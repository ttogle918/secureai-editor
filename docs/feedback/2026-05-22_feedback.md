# 투자 평가 및 기술 코칭 피드백: SecureAI

> **작성일**: 2026-05-22  
> **평가자**: 엑셀러레이터 평가팀 & 기술 코치

---

## 1. 프로젝트 총평 및 투자 매력도 (Investment Potential)

- **주제 부합성**: 매우 높음. 최근 DevSecOps와 AI를 결합한 보안 자동화 도구에 대한 시장의 수요가 폭발적으로 증가하고 있습니다. AI(Claude Haiku)를 활용해 소스코드 취약점을 분석하고(SAST), 패치를 제안하며, 동적 분석(DAST)까지 샌드박스에서 수행하려는 기획은 B2B SaaS 혹은 엔터프라이즈 보안 솔루션으로서 훌륭한 가치를 지닙니다.
- **현실성 (Feasibility)**: 우수함. 문서화된 아키텍처 결정(ADR)과 스프린트 진행 상황을 보면, 단순한 아이디어를 넘어 시스템 격리, 성능 한계, 네트워크 보안, Rate Limit, 분산 처리 등 실제 서비스 운영 시 겪게 될 허들들을 미리 파악하고 설계에 반영했습니다(예: 모듈러 모놀리스 선택, AI 에이전트와 백엔드 분리, SSE 실시간 스트리밍). 소규모 팀의 역량을 고려한 현실적인 기술 스택(Spring Boot + FastAPI) 선택도 돋보입니다.
- **투자자 관점 (결론)**: **매우 긍정적 (Strong Buy / Invest)**. 당장 투자를 집행해도 좋을 만큼 기술적 성숙도와 로드맵이 탄탄합니다. 기술 부채를 통제하는 능력(ADR 작성, 스프린트 관리)이 뛰어나 팀의 실행력에 강한 신뢰가 갑니다.

## 2. 사용자에게 주는 가치 (Value Proposition)

- **개발 보안의 허들 감소**: 전문 보안 지식이 없는 개발자도 코드 작성/커밋 단계에서 즉각적으로 취약점을 인지하고, AI가 제안하는 패치 코드를 통해 빠르게 문제를 해결할 수 있습니다.
- **실시간 피드백 (SSE)**: 분석이 완료될 때까지 기다리지 않고 실시간으로 취약점 발견 과정을 시각화(SSE 스트리밍)함으로써 사용자 경험(UX)을 극대화했습니다.
- **컴플라이언스 대응 자동화**: SBOM 생성 및 CISO, ISMS-P 등 보안 문서 자동 생성 기능(TASK-MISC-002)은 기업 고객들의 페인포인트를 정확히 짚어낸 핵심 셀링 포인트입니다.

## 3. 기술 아키텍처 평가 및 칭찬할 점 (Strengths)

- **전략적 하이브리드 아키텍처**: 모든 것을 MSA로 쪼개지 않고, 트랜잭션이 중요한 비즈니스 로직은 Spring Boot 기반의 모듈러 모놀리스로 유지하면서, 파이썬 생태계가 필수적이고 리소스 격리가 필요한 AI 에이전트만 FastAPI로 분리한 것은 매우 훌륭한 판단입니다.
- **보안/격리 설계 (Trust Boundary)**: DAST 샌드박스를 Docker-in-Docker 대신 Socket 마운트 및 네트워크 격리(dast-isolated-net)로 구현한 점, 메모리에 Access Token을 유지하고 Refresh Token을 HttpOnly 쿠키로 관리하는 전략은 보안 제품에 걸맞은 높은 수준입니다.
- **캐싱 및 성능 최적화**: 파일 해시(SHA-256) 기반 SAST 캐싱으로 AI API 호출 비용과 분석 시간을 획기적으로 줄인 설계는 서비스의 비즈니스 마진을 높이는 데 크게 기여할 것입니다.

## 4. 추가/수정해야 할 사항 (Areas for Improvement & Risks)

- **AI 환각(Hallucination) 리스크 관리**: Claude Haiku 모델이 잘못된 패치나 존재하지 않는 취약점을 제안할 가능성(False Positive/False Negative)에 대한 방어/검증 로직이 다소 부족해 보입니다. AI의 분석 결과를 사용자가 채택하기 전에 검증할 수 있는 단위 테스트 자동 생성 및 실행 기능이 추후 추가되면 좋겠습니다.
- **DAST 샌드박스 자원 관리**: Sprint 6에 예정된 DAST 샌드박스 컨테이너의 경우, 여러 사용자가 동시에 DAST를 요청하면 호스트 노드의 자원(CPU/Memory)이 고갈될 위험이 있습니다. 큐잉 시스템(예: RabbitMQ)이나 동시 실행 컨테이너 수 제한 로직이 더욱 고도화되어야 합니다.
- **토큰 비용 통제**: 엔터프라이즈 플랜(무제한)의 경우 무분별한 분석 요청으로 인해 Anthropic API 비용이 기하급수적으로 늘어날 수 있습니다. 1일/1월 단위의 토큰 사용량 상한(Hard Limit)과 모니터링 알람 시스템이 필수적입니다.

## 5. 배포 및 인프라 전략 조언 (Deployment Strategy)

- **현재 (Docker Compose)**: 초기 검증 및 단일 서버 배포용, SaaS 초기 모델로는 충분합니다.
- **중장기 (SaaS 확장을 위한 K8s)**:
  - B2B Enterprise 고객은 On-Premise 또는 Private Cloud 구축을 요구하는 경우가 많습니다. 향후 Helm Chart 패키징을 통해 고객사 망분리 환경에 손쉽게 배포할 수 있는 구조를 염두에 두어야 합니다.
  - 퍼블릭 SaaS 스케일아웃 시, DAST 실행 노드와 일반 API 처리 노드를 물리적(또는 노드 풀)으로 분리하여 악의적인 DAST 워크로드가 메인 서비스에 영향을 주지 않도록 구성하세요.
- **CI/CD 파이프라인 연동**: VSCode Extension(Sprint 9) 외에도 GitHub Actions, GitLab CI/CD 파이프라인의 플러그인 형태로 동작하게 만들어, PR 생성 시 자동으로 리뷰 코멘트가 달리도록(이미 설계된 `create_issue_comment` 툴 활용) 발전시키면 바이럴과 채택률이 크게 높아질 것입니다.

## 6. 투자 가능성을 더 높이기 위한 조언 (Coaching for Next Level)

1. **타겟 고객군(GTM)의 세분화**: 현재 구현된 기능이 방대합니다(SAST, DAST, SBOM, 모니터링 등). 초기 런칭 시점에는 "중소규모 스타트업의 ISMS-P 인증 통과를 돕는 AI 보안 자동화 도구"와 같이 아주 뾰족한 타겟팅이 필요합니다.
2. **AI 에이전트의 차별화 강조**: 기존 정적 분석 도구(SonarQube, Checkmarx 등)와의 차별점을 영업 자료에서 명확히 부각해야 합니다. 단순한 룰 기반 검사가 아니라, 프로젝트의 문맥과 API Call Chain을 이해하여 비즈니스 로직 상의 취약점(예: IDOR, SSRF)까지 잡아내는 LangGraph 기반의 추론 파이프라인이 핵심 해자(Moat)임을 강조하십시오.
3. **비즈니스 지표 대시보드 마련**: 투자자는 기술 스택뿐만 아니라 비즈니스 모델을 봅니다. 관리자 측면에서 "토큰 소모량 대비 MRR", "고객당 평균 분석 횟수"를 파악할 수 있는 운영 지표 대시보드를 구축하시길 바랍니다.

## 7. 총평
> **"뛰어난 기술적 깊이와 체계적인 프로젝트 관리 역량이 돋보입니다. 개발팀의 기술적 역량은 이미 증명되었으니, 이제는 핵심 가설(시장이 이 AI 패치 제안을 신뢰하고 비용을 지불할 것인가?)을 실제 고객(Beta Tester)을 통해 빠르게 검증하는 단계로 넘어가야 합니다. 긍정적으로 투자를 검토하겠습니다."**
