# SecureAI 검증 확장 — CVE / 비교 테스트 프롬프트 + 검증 로드맵

> 목적: 벤치마크(합성 데이터)만으로 부족한 "실세계 효용"과 "창업자 실력" 근거를 만들고,
> 이를 **단계별 로드맵**으로 묶어 심사·투자에서 "결핍"이 아니라 "추진 중인 계획"으로 보이게 한다.

---

## 1. Claude Code 프롬프트 — 실제 CVE 재현 & 취약점 헌팅

> 레포 루트에서 Claude Code에 그대로 붙여넣기.

```text
SecureAI 모노레포에서 작업해줘. 내 SAST 엔진이 "실제 세상의 진짜 취약점"을 잡는지 검증하고 싶어. 두 가지를 만들어줘: (1) 알려진 CVE 재현 테스트, (2) 실제 OSS 대상 취약점 헌팅 후보 리스트. 프로덕션 코드는 수정하지 말고, 전부 apps/ai_engine/benchmarks/cve/ 아래 독립 하니스로.

[Step 1 — 엔진 진입점 파악]
api/routes/analyze.py, agent/pipeline/security_audit_graph.py, agent/nodes/sast_node.py, agent/nodes/vuln_classifier.py 를 읽고, 단일 파일/레포에 SAST를 돌려서 "탐지된 취약점 목록 + CWE ID + 파일/라인"을 받아오는 가장 깔끔한 방법을 찾아 먼저 보고해.

[Step 2 — CVE 재현 데이터셋 구성]
Java/Python 위주로, 공개된 CVE와 수정 커밋이 명확한 OSS 취약 버전 케이스를 모아줘. GitHub Security Advisory(GHSA) 또는 CVEfixes/Big-Vul 같은 공개 데이터셋을 활용해, 각 케이스에 대해 레포·취약 커밋 해시·취약 파일/함수·CVE ID·CWE를 메타데이터로 정리. 우선 20~30개 소규모로.

[Step 3 — 재현 실행 & 채점]
각 취약 버전에 엔진을 돌려, "그 CVE의 CWE와 일치하는 취약점을 해당 파일 근처에서 신고했는가"로 hit을 판정. 실제 CVE 탐지율(real-world recall)을 전체/CWE별로 계산.

[Step 4 — 취약점 헌팅 후보 생성]
인기 있고 라이선스가 허용되는 공개 Java/Python OSS 레포 몇 개를 골라 엔진을 돌리고, 발견 항목을 심각도·신뢰도 순으로 정렬한 트리아지 리스트(레포/파일/라인/CWE/의심 근거)를 만들어줘. 중복 제거. 이건 자동 제보가 아니라 내가 수동 검증할 후보 목록이야.

[Step 5 — 리포트]
benchmarks/cve/results/ 에:
(a) cve_reproduction.csv — CVE별 탐지 성공/실패
(b) reproduction_scorecard.md — 실제 CVE 탐지율 요약(실행일·모델·케이스 수 포함)
(c) hunting_candidates.md — 수동 검증용 후보 리스트

[제약 — 윤리]
- 정적 분석은 "공개된 소스 코드"에만 수행. 실행 중인 시스템에 대한 공격/능동 스캔 금지.
- 발견을 외부에 알릴 때는 반드시 각 프로젝트의 보안 정책에 따른 coordinated disclosure(메인테이너에게 비공개 제보 후 유예기간)를 따른다. 하니스는 후보만 생성하고, 실제 제보 여부는 내가 수동 판단.
- API 키 커밋 금지(ANTHROPIC_API_KEY는 기존 env). 신규 코드는 benchmarks/ 아래만. README 포함.
- Step 1 통합 지점과 전체 계획을 먼저 보여주고, 대규모 실행은 내 확인 후 시작.
```

**왜 중요한가:** 합성 벤치마크는 "표준 테스트를 통과한다"를, 실제 CVE는 "현장에서 문제됐던 진짜 버그를 잡는다"를 증명한다. 그리고 헌팅으로 미발견 취약점을 찾아 책임 있게 제보하면 **CVE 크레딧**이 남는데, 이건 무경력·무조력 창업자가 *혼자* 만들 수 있는 가장 강력한 신뢰 자산이다. (도구 검증 + 본인 실력 증명을 동시에 해결)

## 2. Claude Code 프롬프트 — 기존 도구 비교 테스트

```text
SecureAI 모노레포에서 작업해줘. 내 SAST 엔진을 기존 표준 도구와 "같은 코드"에서 비교해 우위/상호보완을 표로 만들고 싶어. 프로덕션 코드 수정 금지, benchmarks/comparison/ 아래 독립 하니스로.

[Step 1] 코퍼스 재사용: 앞서 만든 OWASP Benchmark(Java) 결과와 CVE 재현 케이스를 그대로 쓴다.

[Step 2] 비교 도구: 우선 Semgrep(무료 OSS)을 설치해 같은 코퍼스에 돌려줘 (pip install semgrep → semgrep scan --config auto --json). 가능하면 CodeQL도 옵션으로(설정 필요 시 안내). 각 도구의 출력을 (파일, 라인, CWE) 튜플로 정규화.

[Step 3] 채점: 라벨된 벤치마크에서 도구별 TPR/FPR/F1/Youden을 각각 계산. 그리고 차등 분석 — SecureAI가 잡고 Semgrep이 놓친 케이스 / 그 반대 케이스를 따로 목록화.

[Step 4] 리포트: benchmarks/comparison/results/ 에 comparison_scorecard.md(도구별 지표 표 + "SecureAI 단독 탐지 N건" 같은 차등 요약)와 비교 막대그래프(png).

[제약] 동일 코퍼스·동일 판정 기준으로 공정 비교. API 키 커밋 금지. README 포함. 계획 먼저 보고.
```

**왜 중요한가:** 비교 대상이 신뢰받는 도구(Semgrep/CodeQL)라서, "Semgrep이 놓친 N건을 SecureAI가 잡았다"는 한 줄 자체가 외부 레퍼런스가 된다. 단독 수치보다 *비교 우위*가 설득력이 크다.

## 3. 검증 로드맵 (IR/지원서용 — SecureAI 맞춤)

핵심은 **"완료 / 진행 중 / 계획"을 명확히 표시**하는 것. 완료 항목이 있어야 계획이 믿긴다.

### 제품 검증 트랙

| 단계 | 무엇을 증명 | 방법 | 비용/기간 | 신뢰도 | 상태 |
|---|---|---|---|---|---|
| **0** | 탐지 정확도(표준) | OWASP Benchmark + NIST Juliet | ~무료 / 며칠 | 중 (업계 표준) | ▶ 즉시 착수 |
| **0.5** | 탐지의 *질* (탐지만으론 부족) | **CWE 커버리지 매트릭스**(VAL-11) + **결정성/안정성**(VAL-10) | ~무료 / 0단계와 동시 | 중상 | 계획(신규) |
| **1** | 실세계 효용 + 대비 우위 | 실제 CVE 재현(VAL-7) + Semgrep/CodeQL 비교(VAL-8, SARIF 정규화 VAL-13) | 낮음 / 1~2주 | 상 | 계획 |
| **1.5** | **교정** 능력(차별점) | **패치 유효성 검증**(VAL-9): 소거율 + 기능 무회귀 + human fix 대조 | 낮음 / 패치 스프린트와 동반 | 매우 상 (경쟁사 미보유 수치) | 계획(신규·핵심) |
| **2** | 실세계 증명 + 시장 검증 | OSS 취약점 발견 → coordinated disclosure → **CVE 획득** / 파일럿 1~3팀 확보 | 시간 / 1~2개월 | 매우 상 | 계획 |
| **3** | 플랫폼 자체 보안 | 외부 펜테스트 + 자가 도그푸딩 + **분석기 적대적 견고성**(VAL-12, 프롬프트 인젝션 우회 측정) | 중 / 1~3개월 | 상 | 계획 |
| **4** | 엔터프라이즈 신뢰 | GS인증(TTA) → CC인증(공통평가기준) → SOC2 / ISO 27001 | 높음 / 6~12개월+ | 최상 (B2B 필수) | 사업화 단계 |

### 창업자 역량 트랙 (병행)

| 항목 | 상태 |
|---|---|
| 정보보안기사 **필기 합격** | ✅ 완료 |
| 정보보안기사 실기 | ⏳ 진행 중 |
| 본인 도구로 획득한 CVE 크레딧 | 계획 (Stage 2 연동) |
| 보안 커뮤니티/어드바이저 네트워크 | 계획 (OWASP Korea, 보안 밋업 등 / 벤치마크 수치를 들고 접촉) |

### 단계 배치 원칙
- **싸고 빠르고 혼자 가능한 것부터** (0 → 1 → 2). 자본·인맥이 필요한 것(3, 4)은 뒤로.
- 각 단계에 **"프로그램이 어떻게 가속하는가"**를 한 줄씩 붙인다. 예: *"Anthropic·OpenAI 크레딧으로 Stage 0~1을 전체 규모로 실행", "커뮤니티 네트워크로 Stage 2 파일럿 팀 확보"*. → 멘토/프로그램을 "결핍 보충"이 아니라 "레버리지"로 위치시킴.

## 4. 로드맵이 갖는 의미 (요약)

1. **결핍 → 계획.** "인증 없음"이 아니라 "인증까지 가는 단계 + 이미 끝낸 단계". 투자자는 궤적에 베팅한다.
2. **심사관이 의심한 역량을 역증명.** 사업화 기준(인증)을 알고 길을 설계해뒀다는 것 = 자격증 없이 보여주는 *판단력*.
3. **프로그램의 효용을 설계.** 크레딧·네트워크가 내 로드맵을 가속한다고 명시 → "멘토가 필요하다"의 정반대 포지션.
4. **정직함.** 없는 걸 있는 척하지 않고 현재 위치와 방향을 보여줌 — 보안 제품 신뢰의 핵심.

---

## 5. 권장 실행 순서

1. OWASP Benchmark 돌려 0단계 수치 확보 (앞 가이드의 프롬프트)
2. 위 1번 프롬프트로 CVE 재현 → 실세계 탐지율 확보
3. 위 2번 프롬프트로 Semgrep 비교 → 대비 우위 한 줄 확보
4. 헌팅 후보 수동 검증 → 진짜면 coordinated disclosure로 CVE 추진
5. 위 수치·상태를 로드맵 표에 채워 IR 한 장 + 지원서 검증 항목에 삽입

## 6. 추가 검증 방법 (2026-06-15 보강) — "탐지" 너머의 4가지

> 위 1~5는 *탐지·실세계·비교*에 집중. 아래는 (a) 이 제품의 차별점인 **패치**를 검증하고, (b) LLM 기반 도구 특유의 의심에 답하며, (c) 거의 공짜로 IR 숫자를 만든다. 백로그 EPIC-VAL **VAL-9~13**로 태스크화. 상세는 `SECUREAI_BENCHMARK_GUIDE.md` §8.

| 방법 (백로그 ID) | 무엇을 증명 | 한 줄 셀링 | 비용 | 시점 |
|---|---|---|---|---|
| **패치 유효성 검증** (VAL-9) ⭐ | 탐지가 아니라 **교정** — 패치 적용 후 재스캔 소거율 + 대상 테스트 무회귀 + human fix 대조 | "탐지 취약점의 NN%를 자동 패치로 소거, 테스트 회귀 0건" | 낮음 (벤치 코퍼스 재사용) | 패치 자동화 스프린트(14)와 동반 |
| **결정성/안정성** (VAL-10) | LLM-SAST 1순위 의심("매번 다르지 않나") 해소 — 동일 입력 N회 Jaccard 안정성 | "동일 입력 반복 시 finding 안정성 0.9x" | 낮음 | 0단계와 동시 |
| **CWE 커버리지 매트릭스** (VAL-11) | CWE Top 25 / OWASP Top 10 대비 탐지 가능 범위 | "CWE Top 25 중 NN개, OWASP Top 10 전부 커버" | 매우 낮음 | 0단계와 동시 |
| **SARIF 표준 출력** (VAL-13) | GitHub code scanning 연동 + 공정 비교(VAL-8) 정규화 기반 | "SARIF 2.1.0 표준 준수 — GitHub 보안 탭 연동" | 중 | 비교(3)의 전제 |
| **분석기 적대적 견고성** (VAL-12) | 스캔 대상 코드의 프롬프트 인젝션으로 분석기 우회 가능한지 | "도구 자체가 공격에 견딘다" | 중 | 3단계(자체 보안)와 함께 |

**왜 중요한가:** 0~5단계는 "남이 만든 데이터셋에서 잘 잡는다"까지다. **VAL-9(패치 검증)**는 여기서 한 칸 더 나아가 *"문제를 실제로 고친다"*를 숫자로 증명하는데, 이건 탐지 전용 경쟁 도구가 구조적으로 못 내는 수치라 **차별화의 핵심**이다. VAL-10/11은 0단계 하니스만 있으면 거의 공짜로 따라오는 신뢰 숫자다.
