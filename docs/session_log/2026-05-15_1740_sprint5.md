## 2. 의논 내용 & 결정 맥락

### Reviewer BLOCK #1+2: 웹훅 경로 보안 + InternalKeyAuthFilter

`/webhooks/**` 전체 경로를 `permitAll()`로 열어둔 것이 문제. 실제로 허용해야 하는 건 `POST /webhooks/github` 하나뿐이므로 `HttpMethod.POST` 기준으로 축소.

내부 통신 경로(`/api/v1/internal/**`)에 대한 JWT 우회는 의도된 동작이지만 방어 수단이 없었음. `OncePerRequestFilter`를 상속한 `InternalKeyAuthFilter`를 신설:

```java
// /api/v1/internal/** 경로 → X-Internal-Key 헤더 필수
// 헤더 없거나 불일치 → 401 즉시 반환
// SecurityConfig에서 JWT 필터 앞에 삽입
```

`@Value("${secureai.internal-api-key}")`로 키를 주입해 하드코딩 없이 환경 변수에서 읽음.

### Reviewer BLOCK #3: OrgAnalyticsService 분리

`OrganizationService`가 `JdbcTemplate`을 직접 들고 다중 도메인 테이블(sessions, vulnerabilities, projects)을 쿼리하는 것이 SRP 위반으로 판정.

Full CQRS/ApplicationEvent 전환 대신 동일 org 도메인 안에 `OrgAnalyticsService` 클래스를 추가:
- `JdbcTemplate`은 인프라 컴포넌트로 도메인 Repository가 아니므로 도메인 내부 사용은 허용
- `countSessionsByOrgMembers`, `countVulnsByOrgMembers`, `countProjectsByOrgMembers` 3개 메서드
- 모든 쿼리 파라미터는 `?` 바인딩 (SQL Injection 방어 유지)
- `countProjectsByOrgMembers`: `owner_id IN (...)` 서브쿼리 대신 `projects.org_id = ?` FK 직접 사용 (V024 마이그레이션으로 추가된 컬럼)

### Reviewer BLOCK #4: InvitationController Repository 직접 주입 제거

Controller → Repository 직접 의존 금지 원칙 위반. `TeamInvitationRepository`를 Controller에서 제거하고 `getInvitationInfo(token)` 메서드를 `OrganizationService`에 추가. Controller는 Service만 호출.

### 다중 프로젝트 Workspace — 레이스 컨디션 방지

이력 모달에서 다른 프로젝트 세션을 선택하면 두 가지 흐름이 충돌:
1. `switchProjectFromHistory(newProjectId)` → `projectId` 변경 → `useLoadLatestResults` 트리거 → **자동으로 최신 세션 로드**
2. 모달의 `handleLoad` → **선택된 이전 세션 로드**

두 흐름이 동시에 store에 쓰면 어느 쪽이 이기는지 보장 안 됨.

**해결**: `lockedSessionId` 필드를 store에 추가:
- `switchProjectFromHistory(projectId, sessionId)` → `{ projectId, lockedSessionId: sessionId }` 원자적 설정
- `useLoadLatestResults` → `lockedSessionId`가 있으면 auto-load 스킵 (단, `loadedRef.current = projectId` 표시해 이후 재트리거 차단)
- `setProjectId(id)` → `lockedSessionId: null` 클리어 (사이드바에서 프로젝트 전환 시 정상 auto-load 복귀)

### 가상 파일 트리 (분석 기반)

로컬 폴더를 열지 않은 사용자도 분석 결과에서 파일 경로를 탐색할 수 있도록 취약점 `filePath` 목록으로 트리를 합성:

```typescript
function buildVirtualTree(vulns: Vulnerability[]): FileNode[]
// Record<string, any> 브랜치 타입 — 순환 타입 에러 우회
// 경로 분리 → 재귀 Record 구성 → FileNode[] 변환
// type: 'dir' | 'file' (FileNode 타입 기준, 'folder' 아님)
// 정렬: dir 우선, 알파벳 순
```

`hasLocalWorkspace`(폴더 열림 여부)가 false이고 `hasVirtualTree`(취약점 존재)가 true일 때 자동으로 가상 트리 표시. "분석 기반 파일 트리" 레이블로 구분.

### WorkspaceProject 전역 캐시

`AnalysisHistoryModal`이 마운트될 때마다 `GET /projects` + 프로젝트별 세션 API를 중복 호출하는 문제 방지. `useProjects` 훅이 로드 완료 후 `setWorkspaceProjects(summaries)`로 Zustand store에 저장. 모달은 store에서 읽기만 함. `workspaceProjects`는 localStorage 미영속 (런타임 캐시).
