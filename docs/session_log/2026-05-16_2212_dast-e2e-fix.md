## 2. 의논 내용 & 결정 맥락

### HTTP 500 두 가지 근본 원인
백엔드 `POST /api/v1/internal/dast/execute`가 500을 반환하는 원인이 두 곳이었다.

**원인 1**: `saveInitial()`이 try 블록 밖에 있어 FK 위반 시 예외가 전파됨.  
→ try 블록 안으로 이동, 별도 catch에서 실패 응답 반환.

**원인 2**: catch 블록 내 `saveFailed()`가 스스로 throw하면 원래 예외를 덮어 500 발생.  
→ `safelyPersistFailure()` 래퍼로 persist 실패를 격리.

두 케이스 모두 "DAST 분석은 실패했지만 HTTP는 정상 응답"으로 처리하는 것이 맞다고 결론.

### SSE race condition 해결 전략
DAST 그래프가 1ms 내 완료되면 Redis PUBLISH 시점에 EventSource 연결이 없어 이벤트 유실.  
3계층 방어:
1. `notify_node.py`: SETEX(TTL 300s) + PUBLISH 이중 쓰기
2. `api/routes/dast.py`: subscribe 직후 캐시 키 확인 → 이미 결과 있으면 즉시 반환
3. `VulnDetailPanel.tsx`: `setDastSessionId(dastId)` 를 fetch 전에 호출해 SSE 선구독

세 계층 중 하나라도 작동하면 race condition에서 안전하므로 중복 방어를 유지하기로 결정.

### Windows Docker 소켓 제약
`unix:///var/run/docker.sock`은 Windows Docker Desktop 컨테이너 안에서 마운트 불가.  
`tcp://host.docker.internal:2375` + Docker Desktop TCP API 활성화 방식으로 우회.  
`DockerClientConfig.java`가 `DOCKER_HOST` 환경변수를 읽도록 수정 → Linux(소켓)/Windows(TCP) 자동 전환.

### DAST 세션 ID 불일치 (가장 까다로운 버그)
새로고침 후 DAST 결과가 사라지는 현상의 실제 원인이 이것이었다.

- DAST 실행 시: 프론트에서 `crypto.randomUUID()` 생성 → DB의 `exploit_results.session_id`로 저장
- DAST 결과 조회 시: `useLoadLatestResults`가 SAST 분석 세션 ID(`latest.id`)로 조회
- 두 UUID가 완전히 다르므로 항상 0건 반환

처음에는 `/dast/results/{sessionId}` API를 그대로 쓰고 세션 ID를 맞추려고 했으나,
DAST 세션 ID를 localStorage에 저장하는 방식은 여러 취약점에 DAST를 실행할 경우 관리가 복잡해짐.

**최종 결정**: `POST /dast/results/by-vuln-ids` bulk 엔드포인트 추가.  
vuln ID는 SAST 결과에서 얻으므로 항상 알 수 있고, 세션 ID와 독립적으로 조회 가능.

### JSON 상세 보기 설계
`evidence` 텍스트만 보여주는 것보다 `success`, `payload`, `responseSnippet` 전체를 볼 수 있어야
보안 분석가가 실제 공격 내용을 확인할 수 있음.  
`DastResultDetail` 컴포넌트에 "JSON 보기 ▼" 토글 추가 — 평소에는 접혀 있어 UI 노이즈 최소화.
