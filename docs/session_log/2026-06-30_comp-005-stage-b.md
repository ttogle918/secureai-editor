**해결**:

- 결과: 네트워크 의존 0, 파싱 신뢰도 높음

### 2.4 Dedup 설계 결정
**문제**: 반복 크롤 시 중복 피드 아이템 방지 필요 → unique constraint vs. application-level dedup?

**해결**: **content_hash 기반 중복 판단**
- Hash 계산: `SHA-256(sourceUrl + preview)` → 같은 소스 같은 요약 = 같은 의도의 아이템으로 간주
- Preview: summary 앞 N자, summary null이면 content fallback
- DB: `ComplianceFeedItem.contentHash` → **partial unique index**(WHERE content_hash IS NOT NULL)
  
  **왜 partial?** Stage A 초기 시드는 해시 없이 삽입됨(NULL) → 일반 UNIQUE가면 NULL 행들이 경합해서 실패. Partial index로 해시 있는 행만 인덱싱 → 시드와 신규 크롤이 공존 가능하면서도 신규 크롤끼리는 dedup 보호.
