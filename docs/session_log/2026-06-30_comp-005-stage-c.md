## 2. 구현 상세 (무엇을 만들었는지)

### C-1: Backend 크롤러 확장 (Java)

**변경**:
- 신규 파서 클래스: `FeedDetailHtmlParser` (개별 게시물 상세 페이지 파싱) + `KisaDetailHtmlParser` (KISA 구문) + `PdfTextExtractor` (interface) + `PdfBoxTextExtractor` (PDFBox 3.x 기반 PDF→텍스트)
- 신규 DTO: `DetailFetchResult` (url, title, content, source_url)
- 신규 유틸: `FeedHtmlFetcher.fetchBytes()` (바이트 스트림 반환)
- **FeedSource** 리팩토링: `detailParser`, `listParser` 전략 주입
- **ComplianceFeedCrawler** 오케스트레이션 재작성: 목록 → 상세 페이지 폴링 → PDF 텍스트 추출 → content 채움(최대 50,000자) + content_hash 재계산
- **SSRF 방어**: `ALLOWED_FETCH_HOSTS` (www.kisa.or.kr) host allowlist + href-only(href 속성만 신뢰, `169.254.169.254` 등 내부망 차단)

**의존성**:
- PDFBox 3.x는 기존 모놀릿 `build.gradle.kts`에 이미 선언 → 신규 의존성 없음 (전이의존 재사용)

**테스트** (31건 추가):
- 상세 HTML 파서 11건 (요소 추출, 변수 처리, null 안전성)

- 크롤러 통합 15건 (목록+상세 흐름, SSRF 차단, content_hash 갱신)

## 3. 의논 내용 & 결정 맥락

### RAG 인프라 매핑 (Explore 단계)
기존 `security_guidelines` RAG(fastembed BAAI/bge-small-en-v1.5 384차원, 코사인 `<=>`, IVFFlat)를 끝-끝 매핑:
- **재사용률**: 90%+ (임베딩 수집, 벡터 저장소, 검색 함수 시그니처)
- **채택 설계**: compliance_feed_items에 embedding 컬럼 추가 (옵션A. 별도 테이블 옵션B는 조인 오버헤드로 탈락)

### 두 가지 설계 결정 (사용자 확정)
1. **본문 소스 (Stage B vs Stage C)**:
   - Stage B: 목록만 수집 → content=null (요약/링크만)
   - 옵션A: summary MVP (기존 상태 유지)
   - 옵션B: **본문 전문 수집** (C-1 크롤러 확장) ← **사용자 선택**
   - 근거: 컴플라이언스 피드는 표제만 아닌 규제 세부사항까지 RAG 문맥에 필요

2. **한국어 임베딩 (기존 vs 신규)**:
   - 기존: bge-small-en-v1.5 (영어 전용) → KISA 한국어 피드 적합도 낮음
   - 옵션A: 영어 모델 재사용 (성능 희생)
   - 옵션B: **다국어 모델 BAAI/bge-m3** (1024차원) ← **사용자 선택**
   - 근거: 한국어-영어 크로스랭귀지 검색, 모델 업데이트 대비

### 2단계 분할 & 병렬화
- **순서**: C-1(본문 수집) → C-2(임베딩 생성 소비)
- **병렬화 가능**: C-1은 Java(backend), C-2는 Python(ai_engine) → 파일 겹침 없음 → 동시 구현 단축

- 해결: `ALLOWED_FETCH_HOSTS` allowlist (www.kisa.or.kr 만) + href-only (href 속성 신뢰, src/form-action 무시) + 내부망 IP 차단
- 정합성: TASK-1227 DAST 격리화 권고와 일관성 유지
