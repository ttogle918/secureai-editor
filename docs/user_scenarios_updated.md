# SecureAI 사용자 시나리오 (업데이트) 및 테스트 보완 제언

최신 스프린트(Sprint 13)까지의 코드 변경 사항과 세션 로그를 분석한 결과, **사용자 시나리오가 그대로 유지되지 않고 새롭게 추가/변경된 부분이 있습니다.** 특히 AI 엔진의 신뢰성을 검증하는 파이프라인과, 프론트엔드에서 취약점을 처리하는 사용자 액션이 확장되었습니다.

## 1. 최신 코드로 인해 새롭게 추가/변경된 시나리오

1.  **Phase 3: AST 기반 할루시네이션(가짜 취약점) 검증 시나리오 (VAL-3)**
    *   **변경 사항**: 기존에는 AI가 취약점(SAST)을 찾으면 바로 DB에 저장했지만, 이제는 **중간에 결정론적 검증 레이어(AST 검증)**가 추가되었습니다.
    *   **시나리오**: AI가 존재하지 않는 소스코드 라인에 취약점이 있다고 거짓 보고(가짜 인용)를 하면, Python `ast`, Java `javalang`, TypeScript 정규식 검증기가 이를 찾아내어 **저장 전에 강제로 폐기(Discard)**합니다. 이는 재현성을 확보하고 사용자에게 신뢰성을 증명하는 중요한 엣지 케이스 시나리오입니다.
2.  **Phase 4: 트리아지(Triage) 피드백 및 상태 동기화 시나리오 (MOAT-1)**
    *   **변경 사항**: 사용자가 분석 결과를 단순히 보기만 하는 것이 아니라 직접 "확인(Confirm)", "기각(Dismiss)", "패치 적용(Accept Patch)"을 할 수 있는 피드백 루프가 생겼습니다.
    *   **시나리오**: 사용자가 대시보드의 취약점 상세 패널에서 **"오탐 기각(Dismiss)"** 버튼을 누르면 서버에 `false_positive` 상태로 변경되며 영구적으로 트리아지 이력(`triage_feedback`)이 남습니다. 새로고침을 해도 해당 취약점은 해결된 상태(`fixed` 또는 `false_positive`)로 유지되어 사용자 경험의 연속성이 보장됩니다.
3.  **CI / 평가자 시나리오 (VAL-1, VAL-2)**
    *   **변경 사항**: 일반 유저가 아닌 내부 개발자/평가자(VC 등)를 위한 평가 모드 시나리오가 추가되었습니다.
    *   **시나리오**: 개발자가 `make eval LIMIT=100`을 실행하면 OWASP Benchmark를 바탕으로 TPR/FPR/Score를 자동 채점하며, PR을 올릴 때 이전 성능(baseline) 대비 하락폭이 크면 CI에서 경고(warning)를 발생시킵니다.

## 2. 모든 사용자 시나리오 총망라 (Phase 1 ~ Phase 5)

최신 변경 사항을 반영한 사용자 라이프사이클 5단계 시나리오입니다.

### Phase 1: 계정 보안 및 온보딩 (Auth & Security)
*   **정상 경로**: 가입 -> 이메일 인증 -> 로그인 -> 2FA(TOTP) QR 등록 -> 팀 IP Allowlist 등록 -> 시스템 진입.
*   **예외 경로**: 
    *   2FA 인증 앱 삭제 -> 복구 코드로 진입.
    *   허용되지 않은 외부 IP 대역에서 접근 -> 403 Forbidden.
    *   탈퇴 요청(GDPR) -> 즉시 로그인 차단(소프트 삭제) -> 30일 뒤 스케줄러에 의해 영구 삭제(하드 삭제) 및 알림 이메일 발송.

### Phase 2: 프로젝트 및 자산 연동 (Project Setup)
*   **정상 경로**: 새 프로젝트 생성 -> GitHub Webhook 연동 -> 도메인 소유권 검증(TXT 레코드 통과) -> SBOM(CycloneDX 등) 업로드 -> 기존 CVE 자동 매핑.
*   **예외 경로**:
    *   도메인 TXT 레코드 미스매치 시 DAST 동적 분석 권한 자동 박탈.
    *   지원하지 않는 SBOM 포맷 업로드 시 에러 반환.

### Phase 3: AI 기반 정적/동적 분석 실행 (SAST & DAST)
*   **정상 경로**: 분석 시작 -> AI가 코드 취약점 탐지(SAST) -> **[NEW] AST 파서가 취약점 라인의 실존 여부 교차 검증 (가짜 인용 통과 시에만 DB 저장)** -> MCP 서버를 통해 DAST 컨테이너 실행 -> SSE로 프론트엔드/모바일에 실시간 스트리밍.
*   **예외 경로**:
    *   **[NEW]** AI가 없는 코드를 있다고 환각(Hallucination)한 경우 -> AST 검증기가 이를 반려(Discard)하고 폐기 카운트를 올려 가짜 취약점의 노출을 원천 차단.
    *   타겟 서버 지연 시 Docker 컨테이너 타임아웃 -> SAST 결과로 Fallback.
    *   클라이언트 단절 후 재접속 시 SSE 재연결 성공 및 스트리밍 재개.

### Phase 4: 리뷰, 모니터링 및 트리아지 피드백 (Review & Triage)
*   **정상 경로**: 보안 점수 확인 -> **[NEW] 사용자가 오탐지된 취약점을 "기각(Dismiss)" 하거나 "패치(Accept Patch)" 처리 -> 시스템에 피드백 이력 누적 및 상태 영구 반영** -> 매시 정각 헬스체크/SSL 만료 확인.
*   **예외 경로**:
    *   새벽에 서버 다운 또는 SSL 30일 이내 만료 시 Slack Webhook 및 Android 푸시 알림 발송.
    *   **[NEW]** 권한 없는 타 프로젝트의 취약점 트리아지 조작 시도 시 403 차단.

### Phase 5: 다중 클라이언트 경험 (IDE & Mobile)
*   **정상 경로 (VSCode)**: IDE에서 "Analyze" 명령 시 Problem 탭(Diagnostic)에 실시간 하이라이팅 표출.
*   **정상 경로 (Android)**: 분석 완료 푸시 알림 터치 -> 딥링크 앱 실행 -> 로컬 Room DB에서 오프라인 캐시 리스트 확인 -> PDF 리포트 외부 공유(Intent).

        %% 신규 추가된 사용자 피드백(MOAT-1) 루프
        취약점_결과_열람 --> 사용자_트리아지_피드백 : [NEW] 확인/기각/패치 선택
        사용자_트리아지_피드백 --> 피드백_DB_누적(Append-Only)
        피드백_DB_누적(Append-Only) --> 통합_리포트_생성(PDF)

## 4. 추가 테스트 보완 제언 (최신 코드 기준)

최근 스프린트로 인해 시스템의 신뢰성과 검증 프로세스는 많이 발전했지만, 여전히 다음과 같은 테스트 보강이 필요해 보입니다.

1.  **트리아지 동시성(Race Condition) 한계 테스트**:
    *   두 명의 팀원이 동시에 같은 취약점에 대해 각각 "기각"과 "패치 완료"를 처리하려고 할 때 트랜잭션이 안전하게 롤백되거나 하나의 이력만 남는지(낙관적 잠금 등)에 대한 동시성 테스트.
2.  **AST 파서 언어 확장 한계 테스트**:
    *   현재 AST 파서가 Java와 Python은 네이티브 라이브러리로, TS/JS는 정규식으로 검증하고 있는데, 극한의 난독화(Obfuscation)된 코드나 미니파이(Minified)된 코드에서도 정규식 파서가 에러를 뿜지 않고 부드럽게 우회(fallback)하는지 검증하는 단위 테스트 추가 필요.
3.  **실제 벤치마크 (VAL-1) 파이프라인 부하 테스트**:
    *   2,700개의 벤치마크 샘플을 CI에서 지속적으로 돌리다 보면 외부 LLM의 Rate Limit에 부딪히게 됩니다. Rate Limit 429 응답이 집중될 때 지수 백오프(Exponential Backoff)가 정상 작동하는지 모의 서버를 두고 검증하는 테스트.
